SVC (Supervisor Call) in ARM Cortex-M

> **Part of the ARM Cortex-M Programming Series**

---

In every modern operating system, applications cannot directly access critical hardware resources or execute privileged operations. Instead, they request the operating system to perform these actions on their behalf. On ARM Cortex-M processors, this mechanism is implemented using the **Supervisor Call (SVC)** exception.

Whenever an application wants to:

- Create a task
- Allocate memory
- Start the scheduler
- Enable privileged operations
- Access protected peripherals

it simply executes one instruction:

```c
SVC #0
```

This instruction immediately transfers control to the operating system running in privileged mode.

---

## SVC

It is a software-generated exception. Unlike interrupts triggered by hardware, SVC is generated intentionally by software using:

```c
SVC #number
```

Example:

```c
SVC #0
```

The immediate value (0–255) is called the **SVC Number**.

---

Imagine an unprivileged application trying to:

```c
SCB->VTOR = new_table;
```

or

```c
__disable_irq();
```

These operations require privileged access.

If every application could perform them directly:

- One buggy application could crash the entire system.
- Tasks could disable interrupts forever.
- Memory protection would become useless.
- Security would be impossible.

Therefore Cortex-M separates execution into two modes.

---

### Cortex-M Privilege Levels

There are two execution modes.

| Mode | Description |
|-------|-------------|
| Privileged | Full hardware access |
| Unprivileged | Restricted access |

Applications normally execute in:

```text
Thread Mode
Unprivileged
```

Kernel executes in

``` text
Handler Mode
Privileged
```

To request privileged work:

```
Application
      |
      |
      | SVC
      ▼
Kernel
```

SVC is therefore the controlled gateway between user code and the operating system.

---

### Handler Mode vs Thread Mode

Normal code executes in

```
Thread Mode
```

Exceptions execute in

```
Handler Mode
```

Executing

```c
SVC #5
```

causes the processor to leave Thread Mode.

```
Thread Mode
      |
      | SVC
      ▼
Handler Mode
```

Once the handler finishes:

```
BX LR
```

the processor automatically returns.

---

## SVC, an Exception
Cortex-M treats SVC exactly like other exceptions. It has its own vector table entry.

```text
Vector Table

+----------------------+
| Reset                |
+----------------------+
| NMI                  |
+----------------------+
| HardFault            |
+----------------------+
| MemManage            |
+----------------------+
| BusFault             |
+----------------------+
| UsageFault           |
+----------------------+
| Reserved             |
+----------------------+
| SVCall               |
+----------------------+
| Debug Monitor        |
+----------------------+
| PendSV               |
+----------------------+
| SysTick              |
+----------------------+
```

When SVC executes:

The processor jumps directly to

```
SVC_Handler()
```

---

### SVC Instruction Format

The instruction is

```c
SVC #imm8
```

Example

```c
SVC #0 // or number as per CPU
```

---

### Internally Working

Suppose code executes

```c
MOV R0,#10
MOV R1,#20
SVC #3
```

Processor immediately performs Exception Entry.

It automatically stacks

```text
R0
R1
R2
R3
R12
LR
PC
xPSR
```

Then

```
PC

↓

SVC_Handler()
```

No software stack push is needed. The hardware does everything.

---

### Exception Entry Flow

```
Application

MOV R0,#5
SVC #2

        │
        ▼

Hardware Stacks Registers

        │
        ▼

Load SVC Handler Address

        │
        ▼

Execute SVC_Handler()

        │
        ▼

BX LR

        │
        ▼

Registers Restored

        │
        ▼

Application Continues
```

## SVC Number

Different services require different functionality.

Example:

| SVC Number | Meaning |
|------------|---------|
| 0 | Start Scheduler |
| 1 | Create Task |
| 2 | Delete Task |
| 3 | Allocate Memory |
| 4 | Free Memory |
| 5 | Yield CPU |

Instead of multiple handlers, one handler dispatches based on the SVC number.

---

### Extracting the SVC Number

> Inside `SVC_Handler`,
when the stack is pushed by hardware, following things happen:

During exception entry, the processor stacks

```
PC
```

But that PC points to the instruction **after** SVC.

```
Address

1000 : SVC #5

1002 : Next instruction
        ↑
Stacked PC
```

Therefore:

```
SVC instruction = (Stacked PC) - 2
```

Since Thumb instructions are 16-bit, subtract 2 bytes. The handler reads

```c
svc_instruction = *((uint8_t*)(stacked_pc-2));
```
---

### Example SVC Handler

```c
void SVC_Handler_Main(uint32_t *stack_frame)
{
    uint8_t svc_number;

    uint8_t *pc = (uint8_t *)stack_frame[6];

    svc_number = *(pc - 2);

    switch(svc_number)
    {
        case 0:
            StartScheduler();
            break;

        case 1:
            CreateTask();
            break;

        case 2:
            DeleteTask();
            break;

        default:
            break;
    }
}
```

---

### Assembly Wrapper 

Because the stack pointer can be MSP or PSP, most implementations use a small assembly wrapper.

```c
SVC_Handler
    TST LR,#4
    ITE EQ
    MRSEQ R0,MSP
    MRSNE R0,PSP
    B SVC_Handler_Main
```

This passes the correct stack pointer to the C function.

## RTOS Use Case:
Almost every Cortex-M RTOS uses SVC. Typical usage:

```
Application

↓

SVC

↓

Kernel

↓

Perform privileged operation

↓

Return
```

Examples include:

- Starting scheduler
- Switching privilege levels
- System calls
- Kernel API entry
- Secure resource access

---

### Example: Starting FreeRTOS Scheduler

When

```c
vTaskStartScheduler();
```

is called, internally it eventually executes 

```c
SVC #0
```

The SVC handler:

- Initializes PSP
- Restores first task context
- Switches Thread Mode
- Starts multitasking

Without SVC, the kernel couldn't safely enter privileged execution.

---

## Advantages of SVC

- Provides a secure gateway to privileged operations.
- Separates application code from kernel code.
- Simplifies system call implementation.
- Enables RTOS services without exposing hardware directly.
- Supports user/kernel privilege separation.
- Allows multiple services through a single exception handler using SVC numbers.

## Common Errors:

### Reading the Wrong Address

The stacked PC points **after** the SVC instruction. Always subtract 2 bytes (for the 16-bit Thumb instruction) to locate the SVC opcode.

### Ignoring MSP vs PSP

Depending on where the SVC was triggered, the stacked context may be on the **Main Stack Pointer (MSP)** or **Process Stack Pointer (PSP)**. The assembly wrapper determines which one was active.

### Performing Long Operations

The SVC handler executes in **Handler Mode**. Avoid lengthy or blocking operations; dispatch work quickly and return when possible.

### Mixing Privilege Assumptions

SVC changes execution to **Handler Mode (always privileged)**, but returning from the handler does **not** automatically make Thread Mode privileged. Privilege on return depends on the `CONTROL` register and the operating system's design.

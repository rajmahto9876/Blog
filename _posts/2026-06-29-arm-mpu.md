The Memory Protection Unit (MPU) on Cortex-M is one of the most useful features in embedded firmware development. It gives a way to define memory regions and enforce rules such as read-only access, no-execute permision, and privileged-only access. In simple words, the MPU helps MCU to stop invalid memory accesses before they run into hard-to-debug faults.

## What is an MPU?

An MPU is a small hardware block that checks memory accesses against a set of programmable rules. It does not translate virtual addresses like a traditional MMU. Instead, it acts like a guardrail around physical memory.

When a program tries to access a memory region that violates the configured rule, the Cortex-M core raises a MemManage fault. That fault gives you a chance to catch the error and handle it gracefully.

This is especially helpful in embedded systems because memory bugs are common:

- writing past the end of an array
- corrupting the stack
- executing code from RAM or stack memory
- allowing unprivileged code to access critical peripherals
- accidentally changing flash contents at runtime

## Why MPU matters

The MPU is useful because it adds memory isolation to bare-metal and RTOS-based systems. In a typical embedded application, it does the following:

- protect the stack from accidental corruption
- keep flash memory read-only during normal execution
- allow only privileged code to access control registers
- stop code execution from RAM or from the stack
- isolate tasks in an RTOS so one task cannot access another task's memory

On systems without an MPU, a buggy pointer can easily corrupt memory and cause unpredictable behavior. With an MPU, the fault happens immediately and the CPU can trap it.

## How the MPU works

The MPU works with regions. Each region is a memory block with:

- a base address
- a size
- a set of permissions
- memory attributes such as cacheability and shareability

When a memory access occurs, the MPU checks whether the address belongs to one of the configured regions and whether that access is allowed. If not, the access is blocked and a fault is generated.

A typical MPU configuration looks like this:

- Flash region: executable, read-only
- SRAM region: readable and writable, not executable
- Peripheral region: accessible only in privileged mode
- Stack region: writable only by the current task or privileged context

## Important concept: MPU vs MMU
The difference is simple:

- MPU: memory protection only
- MMU: memory protection plus address translation

An MPU is common in Cortex-M devices because they are usually designed for deterministic real-time behavior. They do not need the full complexity of virtual memory management.

## Cortex-M MPU registers

The MPU is configured through a small set of registers. In Cortex-M, these registers are memory-mapped in the System Control Space.

### 1. MPU_TYPE

This register tells you what the MPU can do.

It usually contains information such as:

- how many regions are supported
- whether separate instruction and data regions are available

It is useful when you want to know the hardware capabilities before programming the MPU.

### 2. MPU_CTRL

This is the main control register.

It is used to:
- enable or disable the MPU
- enable default memory map access for privileged code
- enable the MPU during hard fault and NMI contexts when needed

### 3. MPU_RNR

This register selects which MPU region is about to be configured.

### 4. MPU_RBAR

This register defines the base address of the selected region.

It also contains the region number and a valid bit. In other words, it tells the MPU where the region starts.

To configure a region, following steps are involved:

- select the region with MPU_RNR
- write the base address to MPU_RBAR
- define the region attributes in the next register

### 5. MPU_RASR

This is the most important register for region setup.

It contains the region attributes such as:

- enable bit
- region size
- access permissions
- execute never bit
- memory attributes like cacheability and bufferability
- subregion disable bits

### 6. MPU_RLAR

This register defines the limit address of the region and marks the region as enabled.
It is often used together with the base address register to define the memory window precisely.

## A simple MPU configuration example
Below is a typical Cortex-M style example:

```c
#include <stdint.h>

/* CMSIS-style register access */
#define FLASH_BASE 0x08000000U
#define SRAM_BASE  0x20000000U

void mpu_init(void)
{
    /* 1. Disable MPU before changing configuration */
    MPU->CTRL = 0;
    __DSB();
    __ISB();

    /* 2. Configure region 0: Flash memory, read-only, executable */
    MPU->RNR = 0;
    MPU->RBAR = (FLASH_BASE & MPU_RBAR_ADDR_Msk) |
                (0u << MPU_RBAR_REGION_Pos) |
                MPU_RBAR_VALID_Msk;
    MPU->RASR = MPU_RASR_ENABLE_Msk |
                (8u << MPU_RASR_SIZE_Pos) |
                (6u << MPU_RASR_AP_Pos) |
                (0u << MPU_RASR_XN_Pos);

    /* 3. Configure region 1: SRAM, read/write, not executable */
    MPU->RNR = 1;
    MPU->RBAR = (SRAM_BASE & MPU_RBAR_ADDR_Msk) |
                (1u << MPU_RBAR_REGION_Pos) |
                MPU_RBAR_VALID_Msk;
    MPU->RASR = MPU_RASR_ENABLE_Msk |
                (8u << MPU_RASR_SIZE_Pos) |
                (3u << MPU_RASR_AP_Pos) |
                (1u << MPU_RASR_XN_Pos);

    /* 4. Enable MPU and allow privileged default access */
    MPU->CTRL = MPU_CTRL_ENABLE_Msk | MPU_CTRL_PRIVDEFENA_Msk;
    __DSB();
    __ISB();
}
```

- select region number with MPU_RNR
- define the base address with MPU_RBAR
- define the permissions and size with MPU_RASR
- enable the MPU with MPU_CTRL

## Understanding the code

### Region 0: Flash

This region is configured as executable and read-only. That means:

- normal code execution from flash is allowed
- accidental writes to flash from firmware are blocked

### Region 1: SRAM

This region is configured as read/write and not executable. That means:
- your stack and heap can be used normally
- code execution from SRAM is blocked


## MPU permissions

The access permission bits tell the core whether a region can be:
- read-only
- read/write
- inaccessible
- accessible only in privileged mode

A common design is:
- flash: read-only and executable
- SRAM: read/write and non-executable
- peripherals: privileged-only access

## The enable sequence 
The MPU should be configured in a careful order or an immeditate fault is triggered.
The usual pattern is:

1. disable the MPU
2. program all regions
3. synchronize the pipeline
4. enable the MPU

The barriers `__DSB()` and `__ISB()` are important because they ensure the new MPU configuration is visible to the core before execution continues.

## Common MPU pitfalls

### 1. Region alignment

MPU regions must be aligned to their size. A region of 256 KB must start at an address aligned to 256 KB. If it is not aligned correctly, the configuration may behave unexpectedly.

### 2. Overlapping regions

Regions should not overlap. Overlapping regions can make the MPU behavior confusing and hard to reason about.

### 3. Forgetting to enable the MPU

A region is useless if the MPU is disabled. It is good to verify that `MPU_CTRL` has the enable bit set.

### 4. Not handling MemManage faults

If the MPU blocks an access, the CPU raises a MemManage fault. If the memManage handler is disabled, the fault may move to hard fault.

## Example MemManage fault handler

```c
void MemManage_Handler(void)
{
    while (1)
    {
        __asm volatile("nop");
    }
}
```

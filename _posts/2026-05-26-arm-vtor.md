In ARM Cortex-M mcu, every exception, interrupt, fault, and reset event depends on a structure called the **Vector Table**. The **VTOR (Vector Table Offset Register)** allows the processor to relocate this vector table dynamically. For working in bootloaders, RTOS, firmware updates, or debugging embedded systems, understanding VTOR is essential.

---

# Vector Table
A **Vector Table** is an array of addresses stored in memory.
Each entry contains the address of an interrupt handler or exception handler.
The first two entries are special:

| Index | Purpose |
|---|---|
| 0 | Initial Stack Pointer (MSP) |
| 1 | Reset Handler Address |

The remaining entries contain addresses of exception handlers

| Exception | Description |
|---|---|
| NMI_Handler | Non-maskable interrupt |
| HardFault_Handler | Hard fault |
| MemManage_Handler | Memory protection fault |
| BusFault_Handler | Bus access fault |
| UsageFault_Handler | Illegal instruction or state |
| SysTick_Handler | System timer interrupt |
| IRQ Handlers | Peripheral interrupts |

---

# Typical Vector Table Layout

```c
/*
From linker Script, Top of the Stack or fixed address in some MCU.
To initilize the MSP.
*/
extern uint32_t _estack_start; 

__attribute__((section(".isr_vector")))
void (* const vector_table[])(void) =
{
    (void (*)(void))&_estack_start,
    (void)Reset_Handler,
    (void)NMI_Handler,
    (void)HardFault_Handler,
    (void)MemManage_Handler,
    (void)BusFault_Handler,
    (void)UsageFault_Handler,
    ......
};
```

---
Usually, Most systems place the vector table in Flash memory ,i.e., after reset

| Memory Type | Typical Address |
|---|---|
| Flash | 0x00000000 |
| SRAM | 0x20000000 |

# VTOR

The **VTOR (Vector Table Offset Register)** thelps the Cortex-M core to locate the vector table .Instead of always reading vectors from address `0x00000000`, the processor reads them from the address configured in VTOR.

---

## VTOR Register Location:

For ARM Cortex-M3/M4/M7:

| Register | Address |
|---|---|
| SCB->VTOR | 0xE000ED08 |

---

## VTOR Uses:
VTOR is mainly used when:
- Bootloader jumps to application
- Firmware update systems
- RTOS context isolation
- Running code from RAM
- Multiple firmware images
- Secure/non-secure switching
- Dynamic interrupt remapping

---

## Example Code:
### Scenario
The bootloader must tell the CPU to use the application's vector table.

```text
Flash Layout:
0x08000000  -> Bootloader
0x08010000  -> Main Application
```

The application has its own vector table at:

```text
0x08010000
```

### Setting VTOR

```c
#define APP_ADDRESS 0x08010000
SCB->VTOR = APP_ADDRESS;
```

After the exectution of this, all interrupts will use the application's handlers.

### Jumping to Application
The firmware switch code i.e., working of the code when other firmware(bootloader, secure boot) give control to the new firmware.

```c
#define APP_ADDRESS 0x08010000

typedef void (*pFunction)(void);

void JumpToApplication(void)
{
    uint32_t JumpAddress;
    pFunction Jump;

    // Set MSP from application's vector table
    __set_MSP(*(volatile uint32_t*)APP_ADDRESS);

    // Set VTOR
    SCB->VTOR = APP_ADDRESS;

    // Get Reset Handler address
    JumpAddress = *(volatile uint32_t*)(APP_ADDRESS + 4);

    Jump = (pFunction)JumpAddress;

    // Jump to application
    Jump();
}
```

## CPU and the Vector Table Relation

When an interrupt occurs:
- CPU identifies interrupt number
- CPU calculates vector entry address
- Reads handler address
- Jumps to ISR automatically

```text
//each vector entry is 4 bytes.
ISR_Address = VTOR + (IRQ_Number * 4)
```

### Example Calculation

```text
VTOR = 0x08010000
IRQ Number = 20

Then,

ISR Address = 0x08010000 + (20 * 4)
            = 0x08010050
```
The CPU reads the handler address from that location.

## Alignment
VTOR must be aligned properly.
Most Cortex-M devices require:

```text
128-byte alignment
```

### Incorrect Alignment

```c
Wrong:
SCB->VTOR = 0x08010001;

Correct:
SCB->VTOR = 0x08010000;
```

Misalignment can cause:

- HardFault
- Wrong ISR execution
- Random crashes
- System lockup

---

## Relocating Vector Table to RAM
Sometimes interrupts are dynamically modified.

```c
#define RAM_VECTOR_TABLE 0x20000000

memcpy((void*)RAM_VECTOR_TABLE, (void*)0x08000000, 0x100);

SCB->VTOR = RAM_VECTOR_TABLE;
```
Benefits:

- Dynamic ISR replacement
- Faster interrupt access
- Runtime patching
- RTOS flexibility

## Debugging VTOR Issues
### Common Symptoms

| Symptom | Possible Cause |
|---|---|
| HardFault after jump | Wrong VTOR |
| ISR not executing | Incorrect vector table |
| Random interrupts | Corrupted vector table |
| MCU reset loop | Invalid reset handler |
| Wrong ISR called | Misaligned VTOR |

---

### 1. Check VTOR Value
```c
//Must points to valid memory.
SCB->VTOR
```

### 2. Verify Stack Pointer
Check first vector entry:
```c
//Must point to valid RAM.
*(uint32_t*)APP_ADDRESS
```

### 3. Verify Reset Handler
Second entry:
```c
//Must point to valid Flash code.
*(uint32_t*)(APP_ADDRESS + 4)
```

### 4. Check Thumb Bit
ARM Cortex-M uses Thumb mode only.

LSB must be 1:

```text
0x08010101

NOT:
0x08010100

Otherwise:
UsageFault / HardFault
```

# VTOR and RTOS

RTOS systems rely heavily on interrupts:

- SysTick
- PendSV
- SVC

If VTOR is wrong:

- Context switching fails
- Scheduler crashes
- Tasks stop running

Especially important during:

- Bootloader → RTOS jump
- Secure boot
- OTA firmware update

---

# VTOR in Secure Systems

In ARM TrustZone systems:

- Secure vector table
- Non-secure vector table

Separate VTOR registers may exist.

Example:

| Register | Purpose |
|---|---|
| VTOR_S | Secure vector table |
| VTOR_NS | Non-secure vector table |
